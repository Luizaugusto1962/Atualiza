<objective>
Add a concurrency lockfile guard to `principal.sh` so two simultaneous runs cannot corrupt shared state. Use `flock` on a dedicated fd (numeric fd 9 for Bash 4.0 portability on legacy IsCOBOL/SAV servers) against a local lock file `${SCRIPT_DIR}/.atualiza.lock`, created with `umask 077`/`0600` to preserve the security baseline. Acquire the lock at the start of `_main`; release it in the EXIT trap. A second instance must print a clear "already running" message and exit non-zero.

Purpose: Satisfies LOCK-01 — prevents concurrent executions (which share config/backup dirs and could interleave writes) from corrupting state.

Output: `principal.sh` acquires an exclusive non-blocking lock on startup, blocks a second instance with a clear message, and releases the lock on exit.
</objective>

    1) Module-level lock config (use a NUMERIC fd 9 for Bash 4.0 compatibility — do not use `{FD}>` named-fd syntax):
         _LOCK_FD=9
         _LOCK_ARQUIVO="${_LOCK_ARQUIVO:-${SCRIPT_DIR}/.atualiza.lock}"

    2) `_adquirir_bloqueio()` — creates the lock file (the already-active `umask 077` makes it 0600), tightens perms explicitly, opens fd 9 appended to the file, and takes a NON-blocking exclusive `flock`. On failure (already locked or cannot create) it prints a clear Portuguese error to stderr referencing the lock path and returns 1; it must close fd 9 before returning on the failure path. Logic order:
         - `: >>"${_LOCK_ARQUIVO}" 2>/dev/null || { printf "ERRO: Nao foi possivel criar o arquivo de lock '%s'.\n" "${_LOCK_ARQUIVO}" >&2; return 1; }`
         - `chmod 0600 "${_LOCK_ARQUIVO}" 2>/dev/null || true`
         - `exec 9>>"${_LOCK_ARQUIVO}"`
         - `if ! flock -n 9; then printf "ERRO: O Atualiza ja esta em execucao (lock: %s).\n" "${_LOCK_ARQUIVO}" >&2; exec 9>&- 2>/dev/null || true; return 1; fi`
         - `return 0`

    3) `_liberar_bloqueio()` — releases the lock and closes fd 9, guarded so it is safe to call even if the lock was never taken:
         `flock -u 9 2>/dev/null || true; exec 9>&- 2>/dev/null || true`

    Then modify `_main()` (lines 262-272) as follows:
      - Change the EXIT trap from `trap '_resetando' EXIT` to `trap '_liberar_bloqueio 2>/dev/null || true; _resetando' EXIT` (this releases the lock on ANY exit, including INT/TERM/HUP-triggered exits via the other traps which call `_encerrar_programa` → exit).
      - Immediately AFTER the three `trap` lines and BEFORE `# Inicializar sistema`, insert the guard:
          `if ! _adquirir_bloqueio; then _encerrar_programa 1; fi`

    Do NOT touch the module-loading order, `_inicializar_sistema`, or any other function. `_resetando` is defined in config.sh (loaded during init) and is available by the time the EXIT trap fires in production. (When sourcing principal.sh directly for testing, `_resetando` may be undefined — the test in Task 2 sources without invoking `_main`, so the trap is never set; this is fine.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Lock file creation | The lock file is created in `${SCRIPT_DIR}` (operator-writable); a local attacker could pre-create/symlink it |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03 | Tampering (symlink/TOCTOU on lock file) | `.atualiza.lock` creation (principal.sh) | mitigate | Use `flock` on an open fd (atomic, kernel-level) rather than `test -e`+create, which avoids the classic TOCTOU race. The file is created with `umask 077` already active (principal.sh:18) and explicitly `chmod 0600`, so other local users cannot read/intercept it. A pre-existing symlink to a sensitive file: `: >>file` then `exec 9>>file` opens the link target — to harden further, optionally add `[[ -L "${_LOCK_ARQUIVO}" ]] && return 1` before creation; document this as a follow-up if the threat model requires it. |
| T-01-03B | Denial of Service | stale lock after crash | mitigate | Lock is fd-based and released automatically by the kernel when the process (and fd 9) closes — including on crash/kill. The EXIT trap also releases explicitly. No manual `rm` is required, avoiding a delete-based race. |

Security baseline (`umask 077`, `0600`, config-injection defense, path-validation) is preserved — the lock file is created with 0600 and the change touches only principal.sh init.
</threat_model>
